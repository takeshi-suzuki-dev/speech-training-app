# Azure Speech 向け発音採点 API 設計（PoC）

## 概要

本ドキュメントは、Azure Speech を利用した発音採点 PoC の API 設計を示すものです。

この API の責務は次のとおりです。

- ユーザー音声と参照テキストを受け取る
- 発音を評価する
- 構造化された採点結果をフロントエンドへ返す

この API は**採点専用**です。

音声生成は ElevenLabs ベースの別 API が担当するため、本ドキュメントの対象外です。

-----

## スコープ

本ドキュメントは、アプリケーションにおける発音採点側のみを対象とします。

### 対象範囲

- 採点用の音声アップロード
- 参照テキスト入力
- Azure Speech を用いた発音評価
- 構造化されたレスポンス設計
- 採点結果のフロントエンド表示方針

### 対象外

- 音声合成 / TTS
- ElevenLabs 連携

これらの機能は別 API の責務です。

-----

## 目的

この PoC の目的は、発音評価のための明確な API 構造を定めることです。

この PoC では、次の点を確認します。

- 採点に必要な入力は何か
- どのような採点データを返せるか
- 返却された採点データをフロントエンドでどう表示するか

現時点での目的は、本番向け API を最終確定することではありません。

-----

## エンドポイント

`POST /api/pronunciation/score`

### Content-Type

`multipart/form-data`

-----

## リクエスト設計

### 必須項目

- `audio`
  - 発音採点対象となるユーザー音声ファイル
- `referenceText`
  - 発音評価に使用するお手本テキスト

### 現時点では送らない項目

以下の項目は初期 PoC スコープから意図的に外しています。

- `languageCode`
- `mode`
- `userId`
- その他の任意の採点パラメータ

必要になった段階で追加します。

-----

## レスポンス設計

レスポンスは次の 3 グループに整理します。

1. 採点フィードバックとして確実に使うデータ
1. 今後使う可能性があるデータ
1. デバッグや将来の項目抽出のために保持する Azure の元レスポンス

### レスポンス項目

#### 中核となる採点項目

- `transcript`
- `overallScore`
- `sentenceScores`
- `words`
- `phonemes`

#### 補助的な項目

- `extras`

#### 元データ

- `rawJson`

-----

## レスポンス項目詳細

### `transcript`

Azure Speech が認識した文章です。

これはデバッグ用途だけでなく、システムが実際にどう聞き取ったかを表示する目的でも使います。

### `overallScore`

送信された音声に対する全体の発音スコアです。

### `sentenceScores`

文全体に対するスコア群です。例:

- `accuracy`
- `fluency`
- `completeness`
- `prosody`

### `words`

単語単位の採点結果リストです。

各要素には次のような項目を含められます。

- 単語
- スコア群
- エラー種別
- `offset`
- `duration`

### `phonemes`

音素単位の採点結果リストです。

各要素には次のような項目を含められます。

- 対象単語
- 音素
- スコア群
- 正解 IPA
- 音声から判定された候補音素（取得できる場合）
- `offset`
- `duration`

### `extras`

今後使う可能性がある追加レスポンス項目です。

例:

- 補助メタデータ
- まだメイン UI にマッピングしていない Azure 項目

`extras` は、まだ正式項目に昇格していない値の一時置き場です。  
継続的に使う項目は、名前付きの正式レスポンス項目へ昇格させます。

この項目には、`words` や `phonemes` にすでに保持している timing 情報を重複して入れません。

### `rawJson`

Azure Speech から返ってきた元のレスポンス payload です。用途は次のとおりです。

- デバッグ
- 採点結果の検証
- 将来の項目抽出
- PoC 開発時のバックエンド / フロントエンド比較

-----

## レスポンス例

```json
{
  "transcript": "I want to improve my English speaking.",
  "overallScore": 78,
  "sentenceScores": {
    "accuracy": 80,
    "fluency": 72,
    "completeness": 90,
    "prosody": 68
  },
  "words": [
    {
      "word": "want",
      "scores": {
        "accuracy": 82
      },
      "errorType": "Mispronunciation",
      "offset": 1200000,
      "duration": 350000
    }
  ],
  "phonemes": [
    {
      "word": "want",
      "phoneme": "ɑː",
      "scores": {
        "accuracy": 70
      },
      "expectedIpa": "ɑː",
      "candidates": ["ɑː", "ʌ"],
      "offset": 1450000,
      "duration": 120000
    }
  ],
  "extras": {
    "someUnmappedField": "example"
  },
  "rawJson": {}
}
```

-----

## データの関係

`words` と `phonemes` はどちらもトップレベルの項目として返します。

フロントエンドでは、共通の単語参照を使って音素レベルのデータを対応する単語の下にグルーピングして表示します。

これにより、API レスポンスの明示性を保ちながら、単語中心の UI レイアウトも実現できます。

-----

## フロントエンド表示方針

フロントエンドでは返された採点データをすべて表示する前提ですが、すべてを同じ重みで見せる必要はありません。

### Summary セクション

最初に表示する上位の採点情報です。

- `transcript`
- `overallScore`
- `sentenceScores`

### 単語単位の採点セクション

各単語を 1 つの block として表示します。

各 block には次を含めます。

- 単語
- 単語レベルのスコア
- エラー種別
- その単語に属する音素リスト

### 音素単位の採点セクション

音素詳細は、対応する単語 block の中に表示します。

各音素項目には次を含めます。

- 音素
- スコア群
- 正解 IPA
- 候補音素

### 補助セクション

優先度が低い採点関連データは下部に表示します。

- `extras`
- `rawJson`

可読性のため、これらのセクションは折りたたみ表示にしてよいです。

-----

## UI 構成イメージ

PoC 段階では、凝ったスタイルよりもシンプルで読みやすい構成を優先します。

構成例:

- Summary
- Word blocks
  - 単語の採点情報
  - 音素の採点リスト
- Extras
- Raw JSON

この構成により、単語レベルと音素レベルの採点データの関係を理解しやすくします。

-----

## エラーハンドリング

PoC でも、読みやすい失敗情報を返すべきです。

### 想定するエラーケース

- audio が未指定または空
- reference text が未指定
- Azure Speech リクエスト失敗
- transcript は返るが、一部の採点項目が欠けている

### レスポンス方針

バリデーションや処理失敗時には、次を返します。

- 適切な HTTP ステータス
- 読みやすいエラーメッセージ
- 返す意味がある場合に限り、部分的な採点データ

### エラーレスポンス例

```json
{
  "error": "Pronunciation scoring failed",
  "details": "Azure Speech request timed out"
}
```

より詳細な本番向けエラーモデルは後で導入できます。

-----

## 設計原則

この PoC では次の原則に従います。

- 使える採点データは早い段階で API から返す
- 将来使えるかもしれない採点項目を早すぎる段階で隠さない
- 高度なスタイリングがなくても UI を読みやすく保つ
- レイアウトは実装しながら段階的に改善する
- 検証と拡張のために Azure の元レスポンスを保持する
- 採点責務と音声生成責務を分離する

-----

## 責務の境界

### この API が担当すること

- 発音採点
- 採点結果の変換
- フロントエンドへの採点データ提供

### この API が担当しないこと

- 音声合成
- 生成サンプル音声
- 音声選択

これらの責務は ElevenLabs ベースの別 API に属します。

-----

## 実装メモ

### バックエンド

- Spring Boot が `audio` と `referenceText` を受け取る
- バックエンドから Azure Speech を呼び出して発音評価を行う
- 構造化された採点レスポンス DTO をフロントエンドへ返す

### フロントエンド

- Next.js が `multipart/form-data` を送信する
- 採点レスポンスを読みやすいレイアウトで表示する
- 単語レベルと音素レベルの採点情報をグループ化して見せる

-----

## PoC 完了条件

次を満たしたら、この PoC は成功とみなします。

- バックエンドから Azure Speech の発音評価を呼び出せる
- API が構造化された採点レスポンスを返せる
- フロントエンドで次を表示できる
  - `transcript`
  - `overallScore`
  - 文レベルのスコア
  - 少なくとも 1 つの単語レベル結果
  - 少なくとも 1 つの音素レベル結果
- UI 上で `rawJson` を確認できる
- 単語レベル・音素レベルの採点フィードバックが読みやすいレイアウトで見える

-----

## 今後の改善案

PoC 後の次の候補は次のとおりです。

- score object の型をより厳密にする
- 音素グルーピングロジックの改善
- レイアウトと視覚階層の改善
- 採点結果の説明をより分かりやすくする
- 履歴保存と比較表示
- より広い発音トレーニングワークフローとの統合

音声生成は引き続き別責務として独立して設計します。
