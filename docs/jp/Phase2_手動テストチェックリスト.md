# 手動テストチェックリスト

## 1. 目的

この文書は、Speech Training App を限定公開のポートフォリオデモとして準備する一環として、手動テストのチェックリストを定義するものです。

この段階での目的は、完全な自動テストカバレッジを達成することではありません。  
目的は、主要なユーザーフローが正しく動作すること、および承認済みの採用担当者や面接官に見せられる程度にアプリケーションが安全であることを確認することです。

## 2. テスト方針

この段階では、以下に重点を置いてテストします。

- 主要なユーザーフローに対する手動テスト
- 重要な保護APIに対するバックエンドの統合テスト的な確認
- フロントエンドの最小限のスモークテスト
- 認証、認可、データ所有権の動作確認

この段階では、大規模なユニットテストカバレッジは対象外とします。

## 3. テスト環境

- フロントエンド: Next.js ローカル開発サーバー
- バックエンド: Spring Boot ローカルサーバー
- データベース: Supabase PostgreSQL
- ストレージ: Supabase Storage
- 認証: Firebase / Google Sign-In
- 発音採点: Azure AI Speech
- サンプル音声生成: ElevenLabs

## 4. 前提条件

テスト開始前に、以下を満たしていること。

- フロントエンドがビルドエラーなしで起動できること。
- バックエンドがビルドエラーなしで起動できること。
- Supabase のDBスキーマが最新であること。
- 必要な環境変数が設定されていること。
- Firebase の認証設定が完了していること。
- `app_allowed_users` に少なくとも1人の OWNER ユーザーが存在すること。
- アクセス制御テスト用に、allowlist に登録されていない Google アカウントが少なくとも1つ利用できること。
- プリセットカテゴリと文テンプレートが存在すること。
- プリセットテンプレート用のサンプル音声メタデータが存在すること。

## 5. スモークテスト

### ST-01: フロントエンドが起動する

手順:
1. フロントエンドを起動する。
2. ブラウザでアプリケーションを開く。

期待結果:
- ページがランタイムエラーなしで表示される。
- アカウントまたはログイン領域が表示される。

### ST-02: バックエンドが起動する

手順:
1. バックエンドを起動する。
2. health endpoint にアクセスする。

期待結果:
- バックエンドが正常にレスポンスを返す。
- health endpoint では認証が不要である。

## 6. 認証と allowlist

### AUTH-01: allowlist 登録済み Google アカウントでログインする

手順:
1. アプリを開く。
2. ヒーロー部の「Login with Google」をクリックする(Account欄までスクロールする)、またはAccount欄まで直接スクロールする。
3. 「Continue with Google」をクリックする。
4. allowlist に登録済みの Google アカウントでログインする。

期待結果:
- ログインに成功する。
- 今回がログイン直後であり、かつallowlist許可済みのため、自動的に `/pronunciation` へ遷移する。
- ログイン中のメールアドレスが表示される(`/pronunciation`のアカウントメニューから確認可能)。
- 保護されたアプリ機能を利用できる。

### AUTH-02: allowlist 未登録の Google アカウントでログインする

手順:
1. アプリを開く。
2. `app_allowed_users` に登録されていない Google アカウントでログインする。
3. 保護された機能にアクセスする。

期待結果:
- ユーザーはブロックされる。
- バックエンドは 403 を返す。
- フロントエンドには、アクセス拒否のメッセージが明確に表示される。

### AUTH-03: 未認証状態で保護APIにアクセスする

手順:
1. Authorization header なしで保護APIを呼び出す。

期待結果:
- バックエンドは 401 を返す。
- 保護されたデータは返却されない。

### AUTH-04: 期限切れまたは無効化された allowlist ユーザー

注意: テスト後は必ず元に戻すこと。無効のままにすると、自分自身がアプリにログインできなくなる。

手順:
1. allowlist 登録済みユーザーを無効、または期限切れに設定する。
2. そのユーザーでログインする。
3. 保護された機能にアクセスする。

期待結果:
- バックエンドは 403 を返し、メッセージは "Demo access is inactive or expired." となる。
- フロントエンドにそのメッセージが表示される。汎用の「申請してください」という文言ではない。このユーザーは登録済みであり、申請を促すのは誤りだからである。
- ユーザーは保護された機能を利用できない。

### AUTH-05: Google アカウントが紐付け済みの Firebase UID と一致しない

注意: 変更前に元の `firebase_uid` を控え、テスト後に戻すこと。null に戻しても良い(次回ログイン時に再度紐付けられる)。

手順:
1. allowlist 登録済みユーザーで一度ログインし、`app_allowed_users` のレコードに Firebase UID が紐付けられた状態にする。
2. DB上で、そのレコードの `firebase_uid` を別の値に変更する(allowlist登録済みのメールアドレスが、別のGoogleアカウントで後から再登録されたケースを模擬する)。
3. 元のGoogleアカウントで再度ログインし、保護された機能にアクセスする。

期待結果:
- バックエンドは 403 を返し、メッセージは "This Google account does not match the registered Firebase user." となる。
- フロントエンドにこのメッセージが明確に表示される。
- ユーザーは保護された機能を利用できない。

### AUTH-06: ログイン済み状態でトップページに戻る

手順:
1. allowlist許可済みアカウントで`/pronunciation`にログインした状態で、ヘッダー左上の「SpeechAI」ロゴをクリックする。

期待結果:
- トップページ(`/`)に遷移する。
- `/pronunciation`へ自動的に再遷移**されない**(自動遷移はログイン直後のみ発火する仕様)。
- Account欄に、ログイン中のメールアドレス、「Go to Pronunciation App」ボタン、「Logout」ボタンが表示される。

### AUTH-07: アプリヘッダーからのログアウト

手順:
1. `/pronunciation`または`/history`で、右上のグレーのアバターアイコンをクリックする。
2. 表示されたドロップダウンの「Logout」をクリックする。

期待結果:
- ログアウト前、ドロップダウンにログイン中のメールアドレスが表示されている。
- ログアウトされる。
- トップページ(`/`)に遷移する。

## 8. カテゴリ管理

### CAT-01: 表示可能なカテゴリを読み込む

手順:
1. allowlist 登録済みユーザーでログインする。
2. Practice ページを開く。

期待結果:
- プリセットカテゴリが表示される。
- ユーザー所有のカテゴリが存在する場合、それらも表示される。
- 他ユーザーの非公開カテゴリは表示されない。

### CAT-02: ユーザーカテゴリを作成する

手順:
1. “New” category をクリックする。
2. カテゴリ名と説明を入力する。
3. 保存する。

期待結果:
- カテゴリが作成される。
- カテゴリ一覧に表示される。
- そのカテゴリはログイン中の Firebase UID に所有される。

### CAT-03: ユーザーカテゴリを更新する

手順:
1. ユーザー作成カテゴリを選択する。
2. 名前または説明を編集する。
3. 保存する。

期待結果:
- カテゴリが更新される。
- リロード後も更新後の値が保持される。

### CAT-04: ユーザーカテゴリを削除する

手順:
1. 文テンプレートを持つユーザー作成カテゴリを選択する。
2. カテゴリを削除する。
3. 削除確認ダイアログで確認する。

期待結果:
- カテゴリが表示されなくなる。
- そのカテゴリ配下のテンプレートも表示されなくなる。
- 関連する生成済みサンプル音声キャッシュが削除される。
- 学習履歴は削除されない。
- 削除されたデータがUI上に残って表示されない。

### CAT-05: カテゴリに戻った時のデフォルトテンプレート選択

既知の不具合(2026年7月時点): カテゴリに戻った時、そのカテゴリの先頭(一番古い)テンプレートが選択されるとは限らず、想定と異なるテンプレートが表示されることがある。原因は未特定。`pronunciation/page.tsx`のリファクタ時に修正予定。(以前はカテゴリ削除経由で再現していたが、現在の設計ではカテゴリ削除は他のカテゴリへ自動遷移しなくなったため(CAT-07参照)、通常のナビゲーションで再現する形に変更。)

手順:
1. カテゴリAとBをこの順に作成する。
2. Bを選択したまま(途中で他のカテゴリに切り替えずに)、Bに文テンプレートを2件追加する。
3. カテゴリAを選択し、その内容(空、または別の内容)を表示させる。
4. 再度カテゴリBを選択する。

期待結果:
- Bの**先頭(一番古い、リストの一番上)**のテンプレートが自動選択され表示される。直近で追加したものではない。

### CAT-06: 選択中でないカテゴリを削除しても表示は変わらない

既知の不具合(2026年7月時点): 現在の実装は、選択中のカテゴリを削除した際に一覧の先頭へ無条件に飛ぶ実装になっている(CAT-07/CAT-08参照)。このケース(選択中でないカテゴリの削除)は既に正しく動作しているはずだが、将来の修正で意図せず壊れていないかの回帰確認として収載する。

手順:
1. カテゴリAとBを作成する。Aには最低1件テンプレートを入れる。
2. カテゴリAを選択し、表示中にする。
3. カテゴリB(一度も選択していない)を削除する。

期待結果:
- カテゴリAとその選択中のテンプレートの表示は変わらない。
- カテゴリBが一覧から消える。

### CAT-07: 選択中のカテゴリを削除すると、常に未選択状態になる

設計変更(2026年7月時点): カテゴリ削除は、次/前のカテゴリへのフォールバックを行わなくなった。削除したカテゴリが一覧のどの位置にあっても、選択中だった場合はカテゴリ・テンプレートの両方が未選択になり、ユーザーはカテゴリ一覧から選び直す形になる。

手順:
1. カテゴリA、B、Cをこの順に作成する。
2. カテゴリB(どのカテゴリでも良い)を選択し、表示中にする。
3. カテゴリBを削除する。

期待結果:
- カテゴリが選択されていない。
- テンプレートが選択されていない。
- サイドバー/ボトムシートがカテゴリ一覧表示になる。

### CAT-08: 残っているカテゴリが無くなった場合、未選択状態になる

補足: プリセットカテゴリはallowlist許可済みユーザーには常に表示される(CAT-01参照)ため、テスト環境にプリセットカテゴリが無い場合を除き、通常のUI操作ではこの状態に到達できない可能性がある。到達できない場合は、コードレビューで代替する: カテゴリ一覧が空になった時、エラーを出さずにカテゴリ・テンプレートともに未選択になるフォールバック処理を確認する。

手順:
1. プリセットカテゴリが存在しない環境で、ユーザー作成カテゴリを全て削除する。

期待結果:
- カテゴリが選択されていない。
- テンプレートが選択されていない。
- エラーなく、空/カテゴリ無しの状態がUIに表示される。

### CAT-09: シードカテゴリは編集不可

手順:
1. シードカテゴリ(アプリ同梱のもの。自分で作成したものではない)を表示する。
2. サイドバーでホバーし、モバイルのボトムシートでも開く。
3. そのカテゴリに対して `PUT /api/sentence-categories/{id}` を直接呼ぶ。

期待結果:
- どちらのレイアウトでも、カテゴリに編集ボタンが表示されない。
- API を直接呼んだ場合、500 ではなく 404 で拒否される。
- カテゴリを開いて文を練習することは引き続きできる。

## 9. 文テンプレート管理

### TMP-01: カテゴリごとにテンプレートを読み込む

手順:
1. カテゴリを選択する。

期待結果:
- 選択したカテゴリ内の有効なテンプレートが表示される。
- 削除済みテンプレートは表示されない。

### TMP-02: ユーザーテンプレートを作成する

手順:
1. ユーザーカテゴリを選択する。
2. “New” practice sentence をクリックする。
3. タイトル、本文、難易度を入力する。
4. 保存する。

期待結果:
- テンプレートが作成される。
- 選択したカテゴリ内にテンプレートが表示される。
- そのテンプレート用のサンプル音声メタデータが作成される。

### TMP-03: ユーザーテンプレートを更新する

手順:
1. ユーザー作成テンプレートを編集する。
2. 本文を変更する。
3. 保存する。

期待結果:
- テンプレートが更新される。
- サンプル音声の本文が変更された場合、キャッシュ済みサンプル音声はリセットされる。
- 更新後の本文が採点に使用される。

### TMP-04: ユーザーテンプレートを削除する

手順:
1. ユーザー作成テンプレートを削除する。
2. 削除確認ダイアログで確認する。

期待結果:
- テンプレートが一覧から表示されなくなる。
- 関連する favorite 行が削除される。
- 関連する生成済み音声キャッシュが削除される。
- 学習履歴は残る。

### TMP-05: 選択中でないテンプレートを削除しても表示は変わらない

手順:
1. テンプレートが2件以上あるカテゴリで、テンプレート1を選択する。
2. テンプレート2(選択したことのない別のテンプレート)を削除する。

期待結果:
- テンプレート1の選択・表示は変わらない。
- テンプレート2が一覧から消える。

### TMP-06: 選択中のテンプレートを削除すると、次のテンプレートが選択される(次がある場合)

既知の不具合(2026年7月時点): 現在の実装は、次のテンプレートではなくカテゴリ内の先頭のテンプレートを選択してしまう。`pronunciation/page.tsx`のリファクタ時に修正予定。

手順:
1. テンプレートが3件以上あるカテゴリ(順に1、2、3)で、テンプレート2を選択する。
2. テンプレート2を削除する。

期待結果:
- 一覧上でテンプレート2の直後にあるテンプレート3が選択・表示される。

### TMP-07: 選択中のテンプレートが一覧の最後だった場合、前のテンプレートにフォールバックする

既知の不具合(2026年7月時点): TMP-06と同じ原因。

手順:
1. テンプレートが3件以上あるカテゴリ(順に1、2、3)で、テンプレート3(最後)を選択する。
2. テンプレート3を削除する。

期待結果:
- 一覧上でテンプレート3の直前にあるテンプレート2が選択・表示される。

### TMP-08: カテゴリ内の最後の1件を削除すると未選択状態になる

手順:
1. テンプレートが1件だけのカテゴリで、そのテンプレートを選択する。
2. それを削除する。

期待結果:
- テンプレートが選択されていない。
- カテゴリが空のプレースホルダー状態を表示する。真っ白や壊れた画面にならない。
- プレースホルダーの文言は "No practice sentences yet." である。内部の変数名やフック名が混入していないこと。

### TMP-09: シードテンプレートは編集不可

手順:
1. シードカテゴリを開き、シード文を表示する。
2. サイドバーとモバイルのボトムシートの両方を確認する。
3. その文に対して `PUT /api/sentence-templates/{id}` を直接呼ぶ。

期待結果:
- どちらのレイアウトでも、文に編集ボタンが表示されない。
- 削除操作は編集フォーム内にあるため、到達できない。
- API を直接呼んだ場合、500 ではなく 404 で拒否される。
- 文の選択・お気に入り・再生・採点は引き続きできる。

### TMP-10: テンプレート一覧はログイン中のユーザーに限定される

手順:
1. ユーザーAでログインし、シードカテゴリにテンプレートを作成する。
2. ユーザーB(allowlist 登録済みの別アカウント)でログインする。
3. ユーザーBで同じカテゴリを開く。

期待結果:
- ユーザーBにはシード文は見えるが、ユーザーAのテンプレートは見えない。
- ユーザーAには、シード文と自分のテンプレートの両方が引き続き見える。

## 11. サンプル音声

### AUD-01: プリセットテンプレートのサンプル音声を再生する

手順:
1. プリセットテンプレートを選択する(サンプル音声の生成は、再生ボタンではなく**テンプレート選択時**にトリガーされる)。
2. サンプル音声の再生ボタンをクリックする。
3. Supabase Storageのコンソールで、対応する音声ファイルが実際に存在するか直接確認する。

期待結果:
- 音声が生成される、またはキャッシュから読み込まれる。
- 音声が正常に再生される。
- **Supabase Storageに対応する音声ファイルが実際に存在する**(再生できたことだけで判断しない。生成直後のストリーミング再生はStorageへの保存に失敗していても成功して見えることがある)。
- キャッシュが利用可能になった後、同じテンプレートを再度再生すると速くなる。

### AUD-02: ユーザーテンプレートのサンプル音声を再生する

手順:
1. ユーザー作成テンプレートを選択する(サンプル音声の生成は、再生ボタンではなく**テンプレート選択時**にトリガーされる)。
2. サンプル音声を再生する。
3. Supabase Storageのコンソールで、`users/{所有者のFirebase UID}/templates/{テンプレートID}/roger.mp3`に対応する音声ファイルが実際に存在するか直接確認する。

期待結果:
- 認証が必要である。
- allowlist によるアクセス許可が必要である。
- 所有者は音声を生成・再生できる。
- **Supabase Storageに対応する音声ファイルが実際に存在する**(再生できたことだけで判断しない)。
- 他ユーザーは非公開ユーザーテンプレートの音声にアクセスできない。

### AUD-03: テンプレート更新後に音声キャッシュがリセットされる

手順:
1. ユーザーテンプレートを選択してサンプル音声を生成し、Supabase Storageにファイルが存在することを確認する。
2. テンプレート本文を編集する。
3. 保存する。
4. 再度サンプル音声を再生する。

期待結果:
- 古いキャッシュ済み音声は再利用されない。
- 更新後の本文から新しい音声が生成される。
- 新しい音声ファイルがSupabase Storageに保存され、古いものと置き換わる(または新たに追加される)。

## 14. 既知の制限事項

この段階では、以下を既知の制限事項とします。

- 複数音声オプション対応は、主にデータベースおよび設計レベルで実装されている。機能自体はPhase 3に持ち越し。
- 現在のサンプル音声APIは、まだフロントエンドから voice option identifier を受け取らない。
- UIには、まだ2つのサンプル音声再生ボタンが表示されない。
- 本番環境向けのCORS設定はまだ最終化されていない(現状は`http://localhost:3000`のみ許可)。
- 自動テストの対象は、共有された発音練習コンポーネントと API のエラーメッセージ変換に限られる。ページ・フック・バックエンドには自動テストが無いため、この手動チェックリストが主な回帰テストの役割を担う。
- トライアル申請フォームは現在も `mailto:` でメールクライアントを開く方式であり、Webメール利用者にとっては扱いにくい。AWS SES を用いた `POST /api/trial-request` への置き換えはデプロイ時に対応するため、TRIAL-01 は意図した挙動ではなく現状の挙動を検証している。
