# フロントエンド設計(発音練習ページ)

最終更新: 2026-07-12

## 1. 目的

発音練習ページはアプリ最大の画面であり、単一コンポーネントで約2,600行まで肥大した。本書は、そのコンポーネントが現在どう整理されているか、および変更時に従うべきルールを記録する。記述の対象はコードの現在の形であり、変更の経緯ではない。

## 2. 構成

ページは3つに分割されている。

- **フック** (`frontend/src/hooks/pronunciation/`) が状態と副作用を持つ。JSXは含まない。
- **コンポーネント** (`frontend/src/components/pronunciation/`) がマークアップを持つ。presentational であり、状態は持たない。
- **ページ** (`frontend/src/app/pronunciation/page.tsx`) が両者を配線し、どちらにも属さない判断を持つ。

React に依存しない純粋関数は `frontend/src/lib/pronunciation/` に置く。

## 3. フック

| フック | 責務 |
| --- | --- |
| `useCategoryTemplateManager` | カテゴリ、文、お気に入り、最新スコア、選択状態、サイドバー/シートのビュー、カテゴリと文のCRUD |
| `useRecordingSession` | マイク、録音、WAVエンコード、自動停止タイマー |
| `useSampleAudioPlayer` | サンプル音声の生成、キャッシュ、再生 |
| `useScoring` | 採点の実行と結果 |

### 受動的であること(設計方針)

フックは必要な状態を自分から取りに行かない。必要なものは与えられ、起きたことを報告する。

- 入力は引数で受け取る(`canRecord`、`submit({ audioFile, template, … })`)。
- 失敗は注入された `reportError` で報告する。これにより全フックがページの単一のエラーバナーに書き込み、フックが独自のエラー表示を持たない。
- 結果として生じる処理は注入されたコールバック(`onRecordingStart`)で報告する。フックが自分の所有しない状態を直接操作しない。

これが依存を一方向に保つ。`useScoring` は録音の存在を知らず、`useRecordingSession` は採点の存在を知らない。両方を知っているのはページであり、ページが両者を接続する。

### 呼び出し順の制約

フックは依存順に呼ぶ。既に存在するものしか渡せないためである。

1. `useSampleAudioPlayer` — 何にも依存しない
2. `useCategoryTemplateManager` — player のキャッシュ破棄コールバックを受け取る
3. `useScoring` — 初期化時は何にも依存しない
4. `useRecordingSession` — `useScoring` の `reset` を `onRecordingStart` として受け取る

`reset` を他のフックに渡す場合、または effect の依存配列に入れる場合、その関数は安定した参照でなければならない(`useCallback` の依存配列を空にする)。依存配列を広げるのではなく、functional setState updater を使って実現すること。

### ref と、重要な lint ルール

`react-hooks/refs` が有効であり、レンダリング中の ref への書き込みを禁止している。再購読を避けつつ最新値を読みたいコールバックや props は ref に保持し、**effect 内で**同期する。レンダリング中には決して書かない。既存の全フックがこのパターンに従っており、新規のフックも従うこと。

## 4. コンポーネント

`TemplateCard` と `CategoryCard` は、デスクトップのサイドバーとモバイルのボトムシートの両方から描画される。両レイアウトとも維持する。モバイルは今後も使う対象であり、残骸ではない。

カードは `variant`(`"sidebar" | "sheet"`)を受け取り、2つの面の差異をコンポーネント内部の設定テーブルに閉じ込める。呼び出し側はデータとコールバックを渡すのみで、class 名は渡さず、シェブロンや説明文の表示可否も判断しない。

2つの variant は等価ではない。シートはタッチ面であるためホバーによる表示ができず、表示領域も狭い。したがって `CategoryCard` は、説明文とドリルイン用のシェブロンをサイドバーでのみ表示する。

## 5. ページが持ち続けるもの

ページは、どのフック単独の責務でもない判断を持つ。

- サンプル音声を**いつ**先読みするか(選択の変更時)。
- 文を切り替えるとは**何を意味するか**: 録音と採点結果を破棄すること。
- 全フックが報告先とする、共有のエラーバナー。
- ドメイン上の拒否判断。例えば、文が未選択のときに録音を断ること。デバイス起因の失敗はフックが報告するが、その操作に意味があるかどうかはページの判断である。

## 6. 変更時のルール

- ロジックはフックへ、マークアップはコンポーネントへ。どちらもページには置かない。
- フックは必要なものを受け取る。ページの状態を import しない。
- コンポーネントは必要なものを受け取る。フックを import しない。
- 状態管理ライブラリは導入しない。Phase 3 で状態が十分に増えたら、その時点で再検討する。
- 各ステップは `tsc --noEmit` と `eslint` がクリーンな状態で終えること。
- コンポーネントは Vitest + React Testing Library で担保する(`*.test.tsx` をコンポーネントと同じ場所に置く)。これらは jsdom 上で動作するため、どのボタンがどのハンドラに到達するか、どの variant がどの class を生成するかは証明できるが、レイアウト・ホバー・ブレークポイントは見えない。見た目に関わるものは、手動テストチェックリストに従い、ブラウザで両方の幅を確認して検証する。
